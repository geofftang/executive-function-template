# @interrupt — Hyperfocus Exit / Loop Break

## What this is

An external forcing function for when the user's internal exit capacity is offline. This is not a suggestion to consider stopping — it is a structured intervention that breaks the cognitive lock and forces a deliberate choice.

**Clinical basis:**
- Barkley: ADHD requires externalized executive function at the point of performance
- RF-ERP: Demonstrate control over the behavior through structured start/stop
- OCPD behavioral experiments: Each successful stop-and-continue weakens the completion compulsion
- Autism transition protocols: Allow preview and preparation time before context switch

---

## When to fire

### Behavioral signals (pattern-detected, 2+ signals required):
- **Time-loss admission + continued work:** "I've been on this for hours" and keeps going
- **Repetition without progress:** Same question/analysis rephrased 3+ times, no new information
- **Analysis depth inflation:** Increasing granularity on a low-stakes decision across 4+ turns
- **Intent-action gap on stopping:** User says "I should stop" or "one more thing" but continues
- **Scope granularity escalation:** Questions becoming progressively narrower while decision stakes stay constant

### Exclusions:
- First 60 minutes of a focused session
- Executing a well-scoped plan with clear completion criteria
- Within 30-minute cooldown of last interrupt trigger

---

## Protocol

### Step 1 — State Capture (visible, addresses OCPD completion anxiety)

Show the user their work is safe:

> **State captured:** You were working on: [one-line summary]. Your last output was: [what was produced]. This is saved — nothing is lost by pausing.

The user must SEE this. The completion anxiety ("if I stop I'll lose the thread") has less fuel when state is visibly preserved.

### Step 2 — Name the Pattern (RF-ERP: name it, don't engage it)

Name what you're observing without judgment:

> "You've been on [topic] for [estimate]. The detail level has been increasing but the decision stakes haven't changed."

Or:

> "You've asked a variation of [question] three times. Each answer hasn't resolved it — that's a loop pattern, not a research gap."

Or:

> "You said you should stop 20 minutes ago but you're still going. The exit door is missing — that's neurological, not a choice."

**Vary the phrasing.** Do not use the same intervention language every time — novelty collapse means repeated identical interventions get tuned out.

### Step 3 — Force Choice (structured, preserves agency)

Present exactly three options. Do not ask open-ended questions. Do not ask "how do you feel about stopping?"

> **Choose one:**
> 1. **STOP** — End this thread now. Switch topic or close session.
> 2. **TIMEBOX** — Set a hard boundary: [specific deliverable] or [specific time]. When reached, we stop regardless.
> 3. **CONTINUE** — State what "done" looks like and when the next checkpoint fires. (Must be concrete — "when it feels right" is not a boundary.)

**Require a one-line commitment response before continuing.** Do not proceed without an explicit choice.

### Step 4 — Escalation (second trigger in same session)

If the interrupt fires again in the same session on the same topic:

> **Second interrupt.** CONTINUE is no longer available. Choose:
> 1. **STOP** — End this thread now.
> 2. **TIMEBOX** — [specific boundary]. Final extension.

### Step 5 — Log

Record the decision briefly. If the user chose STOP or TIMEBOX and later successfully resumed, note it:

> "You stopped at [X] last session and picked it up cleanly — the work survived."

This is the OCPD corrective experience: evidence that stopping doesn't destroy the work.

---

## Operating principles

- **Offer frameworks, not open questions.** Binary choices for emotional state — never "how do you feel about stopping?"
- **Name it without judgment.** "The exit door is missing" not "you're obsessing."
- **Vary presentation.** Rotate phrasings across invocations to prevent reflexive dismissal.
- **State capture is mandatory.** The OCPD completion fear overrides the intervention if the user can't see their work is safe.
- **Transition time.** Don't demand instant disengagement — allow the user to find a natural seam ("finish the current thought, then we pause").
- **Don't moralize.** This is neurological, not a character flaw. The user already knows they should stop.
